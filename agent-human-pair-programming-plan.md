````markdown
# Goal

Create Pi skills/extensions to support a turn-based human-agent pair programming workflow using:

- Pi as the agent coordinator
- tmux as the persistent workspace
- Neovim as the human review/edit surface
- git commits as turn boundaries
- `.pi/review.md` as the handoff artifact
- persistent dev/test/browser panes as live feedback sources

The workflow should allow the agent and human to alternate work cleanly until a ticket is completed, then squash the intermediate commits into one final commit.

# Core Workflow

## 1. Start ticket session

Given a ticket from the Beads-style ticketing system:

1. Ensure the worktree is clean or ask before continuing.
2. Create or switch to a ticket branch.
3. Start a tmux session for the ticket.
4. Start persistent panes:
   - Pi agent pane
   - dev server pane
   - Vitest/watch test pane
   - Playwright/browser CLI pane
   - optional review/Neovim pane
5. Record session metadata in `.pi/session-state.json`.

Suggested layout:

```text
tmux session: pi-<ticket-id>

pane/window: agent
pane/window: dev-server
pane/window: vitest
pane/window: playwright
pane/window: review
````

# Pair Programming Turn Model

Each participant’s work should be represented by a commit.

```text
Agent turn
  -> make code changes
  -> generate/update .pi/review.md
  -> commit checkpoint
  -> open Neovim for human review

Human turn
  -> edit code and/or annotate .pi/review.md
  -> commit checkpoint
  -> hand back to agent

Agent turn
  -> inspect human commit
  -> continue
```

During the ticket, the commit history may look like:

```text
agent: checkpoint initial implementation
human: review API shape
agent: address human review
human: final polish
agent: fix test/browser regression
review: final agent review
```

At the end, all intermediate commits should be squashed into a clean final commit.

# Git as the Context Protocol

Use git as the primary source of truth.

Important commands:

```bash
git status --porcelain
git rev-parse HEAD
git show HEAD
git show --stat HEAD
git diff HEAD
git log --oneline --decorate -n 10
```

Conventions:

* The current `HEAD` is the latest completed handoff.
* `git show HEAD` shows the previous participant’s work.
* `git diff HEAD` shows the current participant’s uncommitted changes.
* Each handoff should be committed before the next participant begins.
* Agent commits should be local/checkpoint commits, not final project history.
* Finalization should squash all ticket commits into one clean commit.

# review.md Handoff Artifact

`.pi/review.md` is the shared handoff file.

It should be generated by shell/git commands, not manually reconstructed through agent reasoning.

The file should include:

* ticket ID/title
* goal
* acceptance criteria
* current status
* agent summary
* questions for the human
* changed files
* embedded diff for inline comments
* human notes section
* final instruction section

The human may:

* edit code directly
* add inline comments inside the diff
* add comments under file sections
* write final instructions at the bottom
* commit their changes

After Neovim exits, Pi should read:

```bash
git show HEAD
git diff HEAD
cat .pi/review.md
```

If the human committed their changes, Pi should read:

```bash
git show HEAD
cat .pi/review.md
```

# Review Gate Behavior

Create a skill/extension command that performs a review gate.

Pseudo-flow:

```text
pi_review_gate:
  1. Confirm or create checkpoint commit.
  2. Generate .pi/review.md using git/bash.
  3. Open Neovim in a tmux review pane.
  4. Wait for Neovim to exit.
  5. Read .pi/review.md.
  6. Inspect git status.
  7. If human changes are uncommitted:
       read git diff HEAD
       ask whether to continue or request human commit
     Else:
       read git show HEAD
  8. Summarize human feedback.
  9. Continue only after explicit instruction or clear final notes.
```

Neovim command example:

```bash
nvim .pi/review.md <relevant files>
```

Or focused review:

```bash
nvim .pi/review.md src/foo.ts src/foo.test.ts
```

```text

tmux_spawn_pane

tmux_capture_pane
tmux_wait_for_process_exit
tmux_wait_for_pattern
tmux_open_nvim_review
tmux_list_panes
tmux_restart_pane_with_confirmation
```

Read-only operations are safe:

```bash
tmux capture-pane -t <pane> -p -S -200
```

Disruptive operations require confirmation:

```bash
tmux send-keys C-c
restart dev server
restart tests
kill browser
reset database
git checkout/reset
```

The agent may inspect persistent panes freely, but should ask before disrupting them.

# Persistent Feedback Panes

At session start, Pi should start long-running processes once and reuse them.

Suggested panes:

```text
dev-server:
  npm run dev / pnpm dev / yarn dev

vitest:
  npx vitest --watch
  pnpm vitest --watch
  npm test -- --watch

playwright:
  npx playwright test --ui
  npx playwright codegen
  project-specific browser/test command
```

Pi should capture recent pane output for context instead of restarting commands repeatedly.

Examples:

```bash
tmux capture-pane -t "$SESSION:dev-server" -p -S -300
tmux capture-pane -t "$SESSION:vitest" -p -S -300
tmux capture-pane -t "$SESSION:playwright" -p -S -300
```

The agent should use these panes as live sensors:

```text
dev server pane -> runtime/build errors
vitest pane -> failing tests/watch status
playwright pane -> UI/browser behavior
```

# Beads / Ticket Planning Integration

When creating epics and child tickets, an architect agent should also plan checkpoints.

For each ticket, include:

```markdown
## Planned Checkpoints

### CP1: Design/API checkpoint
Trigger:
Human review focus:
Expected artifact:
- checkpoint commit
- .pi/review.md
- passing/failing status

### CP2: Behavior checkpoint
Trigger:
Human review focus:

### CP3: Final review checkpoint
Trigger:
Reviewer agent:
- read-only by default
```

Rules of thumb:

```text
Small ticket:
  final checkpoint only

Medium ticket:
  mid-ticket checkpoint + final checkpoint

Large/risky ticket:
  design checkpoint + behavior checkpoint + final review
```

Good checkpoint triggers:

```text
API shape decided
schema/model changed
migration added
first vertical slice working
tests pass
browser flow verified
security-sensitive code touched
before final squash
```

Avoid checkpointing after every tiny edit.

# Reviewer Agent Flow

At the end of a ticket, optionally run a second agent as a reviewer.

Inputs:

```text
ticket description
acceptance criteria
final branch diff from base
final .pi/review.md
test output
dev/browser logs if relevant
```

Default reviewer mode should be read-only.

Reviewer should produce:

```text
.pi/reviewer.md
```

It should include:

```markdown
# Reviewer Notes

## Summary

## Potential Issues

## Test Gaps

## Suggested Changes

## Approval Status

- approve
- approve with comments
- changes requested
```

Reviewer should not edit source unless explicitly allowed.

---

# Finalization / Squash

At ticket completion:

1. Confirm tests/checks.
2. Optionally run reviewer agent.
3. Remove or archive `.pi` artifacts unless they should be kept.
4. Squash all checkpoint commits into one final commit.
5. Produce final summary/PR notes.

Possible final cleanup:

```bash
git reset --soft <base-branch>
git restore --staged .pi || true
git checkout -- .pi || true
git commit -m "<clean final ticket message>"
```

Or, if `.pi` artifacts should be excluded entirely, ensure `.pi/` is ignored or removed before final commit.

# Safety / Behavior Rules

* Do not silently overwrite human work.
* Require clean worktree at session start unless explicitly overridden.
* Ask before destructive git operations.
* Ask before restarting/killing persistent panes.
* Commit each participant’s handoff.
* Treat commits as turn boundaries.
* Treat `.pi/review.md` as shared handoff memory.
* Treat git as source of truth.
* Prefer Bash/git-generated artifacts over agent-rewritten summaries.
* Keep checkpoint planning lightweight.
* Squash intermediate commits before final delivery.
